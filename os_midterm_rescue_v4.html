🖥 Lect 1 — OS Intro محاضرة 1
  • →OS = وسيط بين اليوزر والهاردوير. بيوفر abstraction للهاردوير المعقد
  • →Kernel = الجزء اللي بيشتغل طول الوقت في الـ memory (first loaded, last unloaded)
  • →System Call = الطريقة الوحيدة للبرنامج يطلب خدمة من الـ OS (software interrupt)
  • →User Mode (محدود) ↔ Kernel Mode (كامل الصلاحيات). Mode bit = 1 user / 0 kernel
  • ⚠Interrupt = hardware | Trap = software (system call أو exception)
🗂 Lect 2 — OS Types محاضرة 2
  • →Batch: بلا تفاعل مع اليوزر، jobs متجمعة
  • →Multiprogramming: أكتر من process في الـ memory في نفس الوقت، CPU utilization ↑
  • →Time-Sharing: كل process بتاخد time quantum — تفاعلي
  • →Real-Time: Hard (فشل لو تأخر) vs Soft (تدهور الجودة)
  • ⚠Virtual Machine: Process VM (برنامج واحد) vs System VM (OS كامل مثل VMware)
⚙️ Lect 3 — Process محاضرة 3
  • →Process = program in execution. Program (passive) → Process (active)
  • →States: New → Ready → Running → Waiting → Terminated
  • →PCB: State, PID, PC, Registers, Memory limits, Open files, Scheduling info
  • →Context Switch = حفظ PCB القديم + تحميل PCB الجديد = overhead (مش شغل مفيد)
  • ⚠Waiting ≠ مستني CPU. Waiting = مستني I/O أو event
📅 Lect 3 — Scheduling محاضرة 3
  • →TAT = Completion − Arrival | WT = TAT − Burst | RT = First Run − Arrival
  • →FCFS: non-preemptive, convoy effect لو الأول طويل
  • →SJF: optimal average WT. Preemptive version = SRTF
  • →RR: preemptive, TQ صغير = overhead عالي. TQ كبير = FCFS
  • ⚠SJF ممكن يسبب starvation للـ long processes
🧵 Lect 4 — Threads محاضرة 4
  • →Thread = unit of execution داخل process. يشارك: Code, Data, Heap, Files
  • →منفصل: Stack, Registers, PC (خاصة بكل thread)
  • →Many-to-One: blocking واحد = blocking الكل. One-to-One (Windows/Linux): true parallelism
  • →Many-to-Many: أحسن flexibility وconcurrency
  • ⚠User-level threads: OS مش عارفهم → blocking = block الكل
🗄 Lect 5 — Memory محاضرة 5
  • →Logical → MMU → Physical. Physical = Base + Logical (simple) أو Frame×Size+Offset (paging)
  • →Paging: page number = addr÷size, offset = addr mod size, Physical = Frame×size+offset
  • →EAT = α(ε+ma) + (1−α)(ε+2ma). α=hit ratio, ε=TLB time, ma=memory access
  • →First-fit (أسرع) | Best-fit (أقل waste) | Worst-fit (الأسوأ)
  • ⚠Paging → Internal Frag | Variable Partition → External Frag

📋 Operating Systems — Overview

ببساطة:

الـ OS زي المدير بتاع الكمبيوتر. انت بتكلم الـ OS، والـ OS هو اللي بيكلم الهاردوير نيابةً عنك. من غيره كنت هتضطر تكتب كود لكل قطعة هاردوير بنفسك!

📌 امتحانياً ركز على:

  • Resource Allocator: بيوزع موارد الكمبيوتر (CPU, Memory, I/O)
  • Control Program: بيتحكم في تنفيذ البرامج عشان يمنع الأخطاء
  • Kernel: الجزء اللي بيشتغل طول الوقت في الكمبيوتر
  • The ONE program running at all times on the computer

🎯 Goals of OS:

  • User Goal: سهل الاستخدام، سريع، موثوق
  • System Goal: سهل التصميم، فعال، flexible
  • بيوفر abstraction للهاردوير المعقد
  • Protected access to shared resources

❌ الغلط الشائع:

ناس كتير بتقول الـ OS = الـ Kernel. مش صح تماماً! الـ Kernel هو الجزء الأساسي من الـ OS، لكن الـ OS كمان بيشمل System Programs.

ببساطة:

الكمبيوتر عنده وضعين: وضع اليوزر (محدود الصلاحيات) ووضع الـ Kernel (فل صلاحيات). ده عشان تحمي الـ OS من البرامج المتعطبة!

💡 System Call:

لما برنامجك محتاج يعمل حاجة OS-level (زي قراءة ملف أو فتح socket)، بيعمل System Call. ده بيخلي الـ CPU ينتقل من User Mode لـ Kernel Mode، ينفذ الطلب، ويرجع تاني.

❌ Trap vs Interrupt:

Interrupt = بييجي من الهاردوير (زي ضغطة keyboard)
Trap = بييجي من البرنامج نفسه (زي System Call أو exception)

للمستخدم:

  • Program execution: تشغيل البرامج
  • I/O operations: تعامل مع الملفات والأجهزة
  • File system manipulation: إنشاء/حذف ملفات
  • Communications: تواصل بين الـ processes
  • Error detection: اكتشاف الأخطاء

للنظام:

  • Resource allocation: توزيع الموارد
  • Accounting: مراقبة استخدام الموارد
  • Protection & security: الحماية والأمان

🔑 Multiprogramming vs Time-Sharing

الفرق الجوهري:

  • Multiprogramming: الهدف هو زيادة CPU Utilization — ما يفضلش الـ CPU فاضي
  • Time-Sharing: الهدف هو تحقيق التفاعلية — كل يوزر يحس إنه مخصص ليه
  • كلاهم بيحتاج Scheduler

⚠️ Real-Time مهم!

نوعين:

  • Hard Real-Time: لو فاتت الـ deadline = كارثة (طيارات، pacemakers)
  • Soft Real-Time: تأخر بسيط مقبول (video streaming)

📊 Scheduling Queues & Schedulers

Long-term Scheduler

🔀 Multithreading Models

Many-to-One

→

User Threads → 1 Kernel Thread

  • ✅ سريع (لا kernel overhead)
  • ❌ لو واحد بلوك الكل بلوك
  • ❌ مش parallel حقيقي

مثال: Solaris Green Threads

One-to-One

↔

كل User Thread → Kernel Thread

  • ✅ Parallel حقيقي
  • ✅ لو واحد بلوك التاني يكمل
  • ❌ Creating thread overhead

مثال: Windows, Linux

Many-to-Many

⇄

m User → n Kernel (n ≤ m)

  • ✅ Flexible وكفء
  • ✅ أحسن من الاتنين
  • ❌ أصعب تنفيذ

مثال: Solaris (قبل v9)

📄 Paging — الفكرة والتطبيق

ببساطة:

بنكسّر الـ Logical Memory في Pages والـ Physical Memory في Frames. نفس الحجم! الـ OS بيحفظ Page Table تقوله "Page رقم X موجودة في Frame رقم Y".

🔢 Address Translation:

الـ CPU بيولد Logical Address مقسوم إلى:

Page Number (p)
Page Offset (d)
Physical = Frame[p] × Page_Size + d

لو Page Size = 2^n:

  • → Offset = آخر n bit من الـ address
  • → Page number = الـ bits المتبقية
  • → عدد الـ Pages = Logical Size ÷ Page Size

⚡ TLB & Effective Access Time (EAT)

ببساطة:

الـ Page Table في الـ RAM — كل access للـ memory محتاج access أول للـ page table (2× وقت!). الـ TLB (Translation Lookaside Buffer) = Cache سريع للـ page table بيقلل الوقت.

EAT = α × (ε + ma) + (1 - α) × (ε + 2×ma)

α = TLB hit ratio | ε = TLB lookup time | ma = memory access time

مهم جداً

🔒 Synchronization — التزامن

ببساطة:

لما أكتر من process أو thread بيوصلوا لنفس الـ data في نفس الوقت، والنتيجة بتتغير حسب ترتيب التنفيذ. ده اسمه Race Condition.

🔴 مثال كلاسيكي — Counter:

Thread A:

register = counter   // = 5
register = register + 1
counter = register   // = 6

Thread B (interleaved):

register = counter   // = 5 !!!
register = register - 1
counter = register   // = 4 !!!
النتيجة المفروض = 5. النتيجة الفعلية = 4. ده Race Condition! Counter = 5 اتقرأ اتنين مرة قبل ما يتحدث.

🔐 Critical Section — الحل المنطقي

الـ Critical Section هي الجزء من الكود اللي بيوصل للـ shared resource. الحل هو إننا نضمن إن process واحدة بس بتشتغل في الـ critical section في نفس الوقت.

1. Mutual Exclusion

process واحدة بس في الـ CS في نفس الوقت

2. Progress

لو الـ CS فاضية، لازم يتاخد قرار مين يدخل

3. Bounded Waiting

process مش بتستنى للأبد — فيه حد للانتظار

FeatureMutexSemaphore
ببساطة🔒 قفل — واحد بس يدخل🎟 عداد — N يدخلوا
القيمة0 أو 1 (binary)أي عدد صحيح ≥ 0
العملياتlock() / unlock()wait() = P() / signal() = V()
المالك✅ فقط اللي عمل lock يعمل unlock❌ أي thread تقدر تعمل signal

Semaphore — wait() و signal():

wait(S): S-- → if S < 0, block
signal(S): S++ → if S ≤ 0, wake one

Binary Semaphore (0 أو 1) = Mutex تقريباً

⚠️ Busy Waiting (Spinlock):

لما process بتستنى في loop وهي شاغلة الـ CPU. ده بيهدر الـ CPU cycles. الحل = block الـ process وتترجعلها لما تتاح الـ resource.

1. Producer-Consumer

Producer بيضيف items، Consumer بياخدهم. Buffer محدود الحجم. لازم synchronization عشان Buffer مش يتملي أو يفضى وكلهم بياخدوا.

Semaphores: full, empty, mutex

2. Readers-Writers

Readers كتير يقروا مع بعض. Writer واحد بس في الوقت الواحد. الـ Writer لازم ياخد exclusive access.

ممكن يؤدي لـ starvation للـ writers

3. Dining Philosophers

5 philosophers محتاج كل واحد 2 forks يأكل. ممكن يحصل deadlock لو كل واحد شايل شوكة واحدة ومستني التانية.

الحل: asymmetric pickup

❌ الغلط الشائع:

ناس بتفتكر إن الـ Mutex والـ Binary Semaphore نفس الحاجة بالظبط. الفرق: الـ Mutex بس اللي عمل lock هو اللي يعمل unlock. الـ Semaphore أي thread تقدر تعمل signal.

🎯 امتحانياً ركز على:

تعريف Race Condition + Critical Section + 3 شروطها + الفرق بين Mutex وSemaphore + wait()/signal() operations

💀 Deadlock — الطريق المسدود

ببساطة:

Deadlock = كل process بتستنى resource محجوزة عند process تانية بتستنى resource عند الأولى — دايرة مسدودة مفيش منها مفر!

🔴 4 شروط الـ Deadlock (لازم كلها تتحقق):

1. Mutual Exclusion

Resource مش ممكن تتشارك — process واحدة بس بتستخدمها في نفس الوقت

2. Hold and Wait

Process شايلة resource وبتستنى resource تانية

3. No Preemption

Resource مش ممكن تتاخد من process بالقوة — بس تتحرر طوعاً

4. Circular Wait

P1 مستنى resource عند P2، وP2 مستنى resource عند P1 (دايرة)

🛠 طرق التعامل مع Deadlock:

✅ Prevention (المنع)

نلغي واحدة من الشروط الأربعة عشان الـ deadlock ميحصلش أصلاً. مثل: كل process تطلب كل resources من البداية (تلغي Hold&Wait).

⚠️ Avoidance (التفادي)

الـ OS بيستخدم info مسبقة عشان يقرر يدي الـ resource أو لأ. الخوارزمية الشهيرة: Banker's Algorithm.

🔍 Detection + Recovery

خلي الـ deadlock يحصل، اكتشفه، وبعدين حله (terminate process أو preempt resource).

🙈 Ignorance (Ostrich Algorithm)

تجاهل المشكلة! بيستخدمه بعض أنظمة زي Windows/Linux لأن الـ deadlock نادر وتكلفة الحل غالية.

    Assignment Edge: R → P (Resource متحجوزة لـ Process)

💡 القاعدة:

لو في الـ Graph فيه cycle وكل resource عندها instance واحدة → Deadlock مؤكد. لو resources عندها instances أكتر → ممكن مش deadlock.

مثال Deadlock:

P1 → R1 (طالبة R1)
R1 → P2 (R1 عند P2)
P2 → R2 (طالبة R2)
R2 → P1 (R2 عند P1)
↳ Cycle! = Deadlock ✅

P1 مستنى P2 تحرر R1، وP2 مستنى P1 تحرر R2 — دايرة مسدودة!

🏦 Banker's Algorithm — الفكرة الأساسية

الـ Banker's Algorithm بيقرر: هل لو أديت الـ Process الـ resources المطلوبة دي، الـ system هيفضل في Safe State؟

Safe State = في ترتيب ممكن أنفذ فيه كل الـ processes لحد ما تخلص (Safe Sequence موجود).

المتغيرات:

  • • Available: Resources المتاحة
  • • Max: أقصى ما تحتاجه كل Process
  • • Allocation: اللي متحجوز لكل Process دلوقتي
  • • Need = Max − Allocation

الخوارزمية:

  1. ابحث عن Process Need ≤ Available
  2. افرض إنها هتخلص → حرر resources
  3. Available += Allocation بتاعتها
  4. كرر لحد ما تخلص كلهم

النتيجة:

لو لقيت Safe Sequence فيه كل الـ processes → Safe State ✅

لو مش قادر → Unsafe State → ارفض الطلب ❌

🔌 I/O Systems — نظام الإدخال والإخراج

🏗 كيف الـ I/O بيشتغل؟

ببساطة:

الـ I/O system بيتوسط بين الـ CPU والـ I/O devices (keyboard, disk, network...). الـ OS بيوفر uniform interface لكل الـ devices عن طريق الـ Device Drivers.

⚙️ I/O Hardware Components:

Port: نقطة اتصال بين الـ device والـ CPU
Bus: سلك مشترك بيربط الـ components (مثل PCI, USB)
Controller: الجزء الإلكتروني اللي بيتحكم في الـ device
Registers: Data, Status, Control registers في كل device

🚀 DMA (Direct Memory Access):

الـ DMA Controller بيعمل نقل الداتا بين الـ device والـ memory بدون تدخل الـ CPU. الـ CPU بس بتدي الأوامر وبعدين بتعمل حاجات تانية. لما يخلص، الـ DMA بيبعث interrupt للـ CPU.

CPU يدي أمر للـ DMA → DMA ينقل Data → Interrupt للـ CPU → CPU تكمل شغلها

💾 Buffering, Caching, Spooling

📦 Buffering

تخزين مؤقت للداتا في الـ memory بين الـ I/O device والـ process. بيساعد لما في فرق بين سرعة الـ producer والـ consumer.

مثال: Buffer لما تسمع موسيقى online

⚡ Caching

نسخة من الداتا في storage أسرع. لو الداتا في الـ cache، نقرأها منه مباشرة. أسرع من الـ disk بكتير.

Cache hit = سريع | Cache miss = روح للـ disk

🖨 Spooling

تخزين الـ output في disk عشان يتبعت للـ device لما يكون جاهز. مشهور مع الـ Printer.

Print Queue = Spooling

📐 Segmentation — التقسيم المنطقي

🧠 ما هو الـ Segmentation؟

ببساطة:

💡 ليه Segmentation؟

البرمجة بتفكر في units منطقية (functions, arrays, stack). الـ Segmentation بيعكس النظرة دي في الـ hardware — بيدي الـ programmer abstraction حقيقية.

🔀 Segmentation with Paging — الأفضل من الاتنين

ببساطة:

بنجمع مزايا الاتنين: الـ Segmentation بيدي الـ logical view للمبرمج، والـ Paging بيلغي الـ external fragmentation ويسهل الـ memory management.

📐 كيف بيشتغل:

  1. الـ Logical Address = Segment# + Offset
  2. الـ Offset يتقسم: Page# + Page Offset
  3. كل Segment عنده Page Table خاصة بيه
  4. الـ Segment Table بتشاور على Page Table بتاعته
  5. الـ Page Table بتدي الـ Frame Number
  6. Physical = Frame × PageSize + PageOffset

✅ المزايا:

✅ Logical view زي الـ Segmentation (Code, Stack, Heap...)
✅ لا External Fragmentation (بسبب الـ Paging)
✅ Easy Sharing — Segments مشتركة بين Processes
✅ Protection per Segment (read-only code segment)
⚠️ Internal Fragmentation لسه موجودة (من الـ Paging)

🔄 الترجمة الكاملة:

Seg# → Seg Table → Page Table Address → Page# → Page Table → Frame# → Physical = Frame×Size + Offset

مسألة 3

Paging Address Translation

📋 المعطيات:

  • Page Size = 4 bytes
  • Logical Address = 13
  • Page Table: [0→5, 1→3, 2→1, 3→7]

🔢 الحل:

  • Page Number = 13 ÷ 4 = 3 (quotient)
  • Offset = 13 mod 4 = 1
  • Page 3 → Frame 7 (من الجدول)
  • Physical = 7 × 4 + 1 = 29

⚠️ مش تنساه:

Page Number = Logical Address ÷ Page Size (بالقسمة الصحيحة، الجزء الصحيح)
Offset = Logical Address mod Page Size
Physical = Frame Number × Page Size + Offset

مسألة 4

TLB & EAT Calculation

📋 المعطيات:

  • TLB hit ratio (α) = 80% = 0.8
  • Memory access time (ma) = 100ns
  • TLB lookup time (ε) = 20ns

📐 الحل:

EAT = α(ε + ma) + (1-α)(ε + 2ma)

= 0.8 × (20 + 100) + 0.2 × (20 + 200)

= 0.8 × 120 + 0.2 × 220

= 96 + 44

= 140 ns

💡 اتفهم المعادلة:

  • TLB hit (α): وقت الـ TLB lookup + وقت access الـ memory مرة واحدة = ε + ma
  • TLB miss (1-α): وقت الـ TLB lookup + access الـ page table + access الـ memory = ε + 2ma

📅 Scheduling Summary

  • 🔵 FCFS: Non-preemptive, Convoy Effect, simple
  • 🔵 SJF: Non-preemptive, optimal avg WT, starvation
  • 🔵 SRTF: Preemptive SJF, best avg WT

  • 🔵 Paging: solves external fragmentation
  • 🔴 Paging still has internal fragmentation

📄 Paging → Fragmentation → TLB → EAT

ليه Paging موجود؟

الـ Contiguous Allocation بيعمل External Fragmentation. الـ Paging بيحلها بتقسيم الـ memory لـ fixed-size pages وframes.

لكن Paging بتعمل Internal Fragmentation (آخر page ممكن مش مليانة)

ليه TLB موجود؟

الـ Page Table في الـ RAM. كل access للـ memory = access للـ page table (overhead). الـ TLB هو cache سريع بيحفظ الـ recent entries.

🔒 Synchronization & Deadlock — الأساسيات

🔄 Synchronization

ببساطة: لما threads كتيرة بتوصل لنفس الـ data في نفس الوقت، ممكن تحصل مشاكل — ده اسمه Race Condition.

Critical Section:

الجزء من الكود اللي بيوصل للـ shared resource. لازم يتحمي بـ Mutex/Semaphore.

Mutex = lock بياخده thread واحد بس في وقت واحد
Semaphore = عداد بيسمح بـ N threads في نفس الوقت

💀 Deadlock

ببساطة: كل process مستنية resource محجوز عند process تانية — دايرة مسدودة!

4 شروط لازم كلها تتحقق:

  • Mutual Exclusion: resource مش ممكن تتشارك
  • Hold & Wait: process شايلة resource ومستنية غيرها
  • No Preemption: مش ممكن تاخد resource بالقوة
  • Circular Wait: دايرة انتظار بين الـ processes
