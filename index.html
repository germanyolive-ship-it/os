📖OS Basics — أهم تعريفات
  • 🔸 OS = intermediary بين user وال hardware
  • 🔸 Kernel = القلب الحقيقي لل OS، اللي بيتحمل في memory طول الوقت
  • 🔸 OS ≠ Kernel — OS أكبر، Kernel جزء منه
  • 🔸 System Call = الطريقة الوحيدة إن الـ user program يطلب خدمة من الـ Kernel
  • 🔸 User Mode → Kernel Mode بيتعمل عن طريق System Call أو Interrupt
  • 🔸 Interrupt = من الـ hardware (خارجي) | Trap = من الـ software (داخلي/user)
  • 🔸 Bootstrap = أول برنامج بيشتغل لما الكمبيوتر يبدأ (في ROM)

⚙️Process — أهم نقاط
  • 🔸 Process = program قيد التنفيذ (active) | Program = ملف ساكن (passive)
  • 🔸 States: New → Ready → Running → Waiting → Terminated
  • 🔸 PCB = Process Control Block — فيه كل info عن الـ process
  • 🔸 Context Switch = لما الـ CPU يتنقل من process لـ process — بيحفظ/يرجع PCB
  • 🔸 Long-term scheduler: disk → memory | Short-term: memory → CPU
  • 🔸 Running → Waiting: بسبب I/O | Running → Ready: بسبب interrupt
📊Scheduling Formulas 🔢
Turnaround Time (TAT):
= Completion Time − Arrival Time
Waiting Time (WT):
= TAT − Burst Time
Response Time:
= First Response − Arrival Time
FCFS: أسرع في تطبيق | SJF: أقل avg WT | RR: أفضل response time

  • 🔸 TLB = cache سريع لل page table

Overview — أساسيات OS

ما هو الـ Operating System؟

يعني إيه؟

الـ OS هو برنامج بيعمل كـ intermediary (وسيط) بين الـ user والـ hardware — بيسهل الاستخدام وبيدير الموارد

كـ Resource Allocator

بيوزع موارد الكمبيوتر (CPU، memory، I/O) بشكل عادل وكفء بين البرامج

كـ Control Program

بيتحكم في تنفيذ البرامج عشان يمنع الأخطاء وسوء الاستخدام

🎯 Goals of OS

  • ✅ تنفيذ برامج الـ user وحل مشاكله
  • ✅ الكمبيوتر يكون سهل الاستخدام
  • ✅ استخدام الـ hardware بكفاءة
  • ✅ حماية البيانات والموارد (Protection)
  • ✅ التواصل بين البرامج المختلفة

الـ Kernel 🔑

ما هو الـ Kernel؟

هو القلب الحقيقي لل OS — أول حاجة بتتحمل في memory وقت الـ Boot وبيفضل موجود طول الجلسة

وظائفه:
  • • Scheduler: بيوزع وقت CPU
  • • Memory Manager: بيدير الـ RAM
  • • Interrupt Handler: بيعالج الـ interrupts
  • • Supervisor: بيدير الصلاحيات
⚠️ مش نفس الـ OS!

الـ OS أكبر — بيشمل Kernel + Utilities + Drivers + User Interface

الـ Kernel بس هو اللي بيشتغل في Kernel Mode

System Calls 📞

يعني إيه؟

الطريقة الوحيدة اللي الـ user program بيستخدمها عشان يطلب خدمة من الـ Kernel (زي فتح ملف، عمل process جديد، إلخ)

🔸 بيتسمى كمان software interrupt

🔸 لما الـ system call بيتنادى → الـ mode بيتغير من User → Kernel

🔸 بعد ما الـ Kernel ينهي الخدمة → بيرجع Kernel → User Mode

⚡ Interrupt vs Trap — فرق مهم جداً

Interrupt (Hardware Interrupt)
  • ✅ مصدره: الـ Hardware (keyboard, disk, timer)
  • ✅ خارجي — من جهاز
  • ✅ بيحصل في أي وقت (asynchronous)
  • ✅ مثال: الـ disk انتهى من قراءة بيانات
Trap (Software Interrupt)
  • ✅ مصدره: الـ Software (برنامج المستخدم)
  • ✅ داخلي — من instruction
  • ✅ بيحصل بشكل متعمد أو بسبب خطأ
  • ✅ مثال: System Call أو Division by Zero

🎯 امتحانياً: كلاهما بيخلي الـ CPU يتحول لـ Kernel Mode ويشغل handler معين

OS Services

User Interface (UI): CLI أو GUI للتواصل مع المستخدم
Program Execution: تحميل وتشغيل البرامج
I/O Operations: التعامل مع أجهزة الإدخال/الإخراج
File System Manipulation: قراءة/كتابة/حذف ملفات
Communication: التواصل بين الـ processes
Error Detection: اكتشاف ومعالجة الأخطاء
Resource Allocation: توزيع CPU, Memory, I/O
Protection & Security: حماية الموارد والبيانات

Booting Process 🚀

1
Power ON → CPU ينشط الـ BIOS/UEFI
2
POST (Power-On Self-Test) → بيفحص الـ Hardware كله
3
بيقرأ الـ MBR (Master Boot Record) من الـ disk
4
Bootstrap Loader (في ROM) بيحمل الـ Kernel في الـ RAM
5
الـ Kernel بيشتغل ويتحكم في كل حاجة
⚠️ الـ Bootstrap program عادة بيتخزن في ROM أو EEPROM — مش في RAM

أنواع الـ Operating Systems

📦

Batch Systems

يعني إيه؟ Jobs بتتجمع وتشتغل واحدة ورا التانية بدون تدخل من المستخدم
العيب: لو job واحدة وقعت في I/O → الـ CPU بتقف (idle) — مش كفاءة!
الميزة: بسيطة جداً في التنفيذ

تاريخياً: كانت بتستخدم punch cards

🔄

Multiprogramming

يعني إيه؟ أكتر من process في الـ memory في نفس الوقت — لما واحدة تعمل I/O، التانية تشتغل
الفايدة: CPU مش بتقف — استخدام أحسن
ملاحظة: مفيش حاجة اسمها "conversation" مع المستخدم
⏱️

Time-Sharing

يعني إيه؟ نوع من الـ Multiprogramming — بس الـ CPU بتتبدل بسرعة بين الـ users عن طريق time slices
الميزة: كل user حاسس إنه عنده الـ CPU لوحده (interactive)
امتحانياً: الفرق بين Multiprogramming وTime-Sharing هو وجود time quantum والـ interactivity
⚡

Real-Time Systems

يعني إيه؟ لازم الـ response يحصل في وقت محدد (deadline) — التأخير ممكن يكون كارثي
أمثلة: طائرات، أنظمة طبية، محطات نووية

نوعان:

• Hard Real-Time: فشل الـ deadline = كارثة

• Soft Real-Time: فشل الـ deadline مش نهاية الدنيا

🌐

Distributed Systems

يعني إيه؟ أكتر من كمبيوتر مترابطين عن طريق network ويشتغلوا كـ system واحد
الميزة: زيادة في الـ reliability والـ performance
📱

Embedded Systems

يعني إيه؟ OS مخصص لجهاز معين — بيعمل مهام محددة بس
خصائص: Kernel صغير + real-time + محدود الموارد
أمثلة: موبايل، سيارة، ميكروويف، راوتر
🖥️

Virtual Machines

يعني إيه؟ Software بيعمل emulation لـ machine كاملة — بتخلي OS يشتغل فوق OS تاني
Process VM: بيعمل environment لـ process واحدة (زي JVM)
System VM: بيعمل environment لـ OS كامل (زي VMware, VirtualBox)
⚠️ امتحانياً: الـ VM بيديك illusion إنك عندك machine منفصلة — في الحقيقة Hardware واحد مقسم
فايدته: تشغيل أكتر من OS على نفس الجهاز — isolation و testing و cloud computing

Context Switch

يعني إيه؟ لما الـ CPU تتنقل من process لـ process تانية — بتحفظ PCB القديمة وتحمل PCB الجديدة
1 Save state of current process في الـ PCB بتاعته
2 Select next process من الـ ready queue
3 Load state of new process من الـ PCB بتاعته
4 Resume execution من النقطة اللي وقف عندها
⚠️ Cost: Context switch بياخد وقت (overhead) — خلال الـ switch، مفيش شغل حقيقي بيحصل

أنواع الـ Schedulers

Schedulerبيشتغل بينوظيفتهسرعة الاستدعاء
Long-term (Job Scheduler) Disk → Memory بيقرر أنهي processes تتحمل في الـ RAM بطيء (دقائق)
Short-term (CPU Scheduler) Memory → CPU بيقرر أنهي process تاخد الـ CPU دلوقتي سريع جداً (milliseconds)
Medium-term Memory ⇄ Disk Swapping — بيشيل processes من memory لو محتاجين مساحة متوسط
امتحانياً: اللي بيتكلم عنه أكتر في الامتحان هو الـ Short-term scheduler (CPU Scheduler) — ده اللي بيعمل الـ FCFS, SJF, RR

Scheduling Queues

Job Queue

كل الـ processes في النظام (disk + memory)

Ready Queue

الـ processes اللي في الـ memory وجاهزة تشتغل في الـ CPU

Device Queue

الـ processes اللي بتستنى جهاز I/O يخلص

📊

CPU Scheduling

Scheduling Criteria — بنحكم على الـ Algorithm بإيه؟

CPU Utilization ↑ نسبة استخدام CPU — أعلى أفضل
Throughput ↑ عدد الـ processes المنتهية في الوحدة الزمنية
Turnaround Time ↓ الوقت من بدء لإنهاء العملية
Waiting Time ↓ الوقت في الـ ready queue
Response Time ↓ وقت أول استجابة للـ request
Fairness ↑ كل process تاخد حظها

Preemptive vs Non-preemptive

Non-preemptive: لما process تاخد الـ CPU، مفيش حاجة تقدر تاخدها منها إلا لو هي خلصت أو طلبت I/O
Preemptive: ممكن تاخد الـ CPU من process وتديها لحد تاني في النص — أعدل وأحسن response time
FCFS

First-Come First-Served

الفكرة: الأول يجي الأول يتخدم — زي طابور عادي
العيب: Convoy Effect — process طويلة ممكن تخلي كل التانيين يستنوا
الميزة: بسيطة جداً في التطبيق
نوع: Non-preemptive فقط
SJF/SRTF

Shortest Job First / SRTF

الفكرة: الـ process اللي Burst Time أقل تشتغل الأول
الميزة: أقل average waiting time نظرياً (optimal)
العيب: Starvation — الـ long processes ممكن تستنى للأبد
SRTF = SJF Preemptive — لو جت process أقل burst time، بتاخد الـ CPU
RR

Round Robin

الفكرة: كل process بتاخد time quantum (q) — لو مخلصتش، ترجع للآخر وتستنى
الميزة: عادل جداً — لا Starvation — أحسن response time
التحدي: لو q صغير → context switches كتير. لو q كبير → بيبقى زي FCFS
نوع: Preemptive

Memory Management (حتى Slide 40 بس)

Logical vs Physical Address

Logical Address (Virtual)

العنوان اللي الـ CPU بيولده — الـ program بيشوفه

مش العنوان الحقيقي في الـ RAM

MMU

Memory Management Unit — Hardware بيحول Logical → Physical

بيضيف Relocation Register على الـ Logical Address

Physical Address

العنوان الحقيقي في الـ RAM

الـ memory unit بتشوفه

Address Binding — متى بتتحدد العناوين؟

Compile Time

لو عارفين مكان الـ process في الـ memory من الأول — الـ compiler بيولد physical addresses مباشرة

Load Time

لما الـ OS بيحمل البرنامج في الـ memory — بيعمل relocatable code

Execution Time ✅ الأشيع

العنوان بيتحدد أثناء التشغيل — بيحتاج MMU hardware — الـ process ممكن تتحرك في الـ memory

Contiguous Allocation وطرق الـ Allocation

Contiguous: الـ process بتاخد block متواصل في الـ memory
First-Fit: خد أول Hole كبير كفاية — أسرع
Best-Fit: خد أصغر Hole كافي — بيقلل الهدر بس أبطأ
Worst-Fit: خد أكبر Hole — بيسيب أكبر Hole متبقي
امتحانياً: First-fit وBest-fit أحسن من Worst-fit في الغالب

Fragmentation — أهم فرق في الميموري

External Fragmentation

المساحة الكلية الفاضية كافية للـ process، بس مش متواصلة (متقطعة)

بيحصل في Contiguous Allocation

الحل: Compaction أو Paging

Internal Fragmentation

المساحة المخصصة للـ process أكبر من اللي هي محتاجاه — الفرق waste

بيحصل في Paging (آخر page مش كاملة)

Paging 📄

الفكرة: الـ Physical Memory بتتقسم لـ Frames (أحجام ثابتة) — الـ Logical Memory بتتقسم لـ Pages (نفس الحجم)
الميزة: بتحل External Fragmentation — الـ process مش لازم تكون متواصلة في الـ memory
العيب: Internal Fragmentation في آخر page
Page Table: جدول بيربط كل page رقم كام بالـ frame اللي فيها
Logical Address = Page Number | Offset Page Number (p) → index في الـ Page Table Offset (d) → مكانها داخل الـ Frame Physical Address = Frame Base + Offset لو Logical Address Space = 2^m و Page Size = 2^n → Page Number = m-n bits → Offset = n bits

🔢 Paging Address Translator

أدخل البيانات وكبس Translate

TLB — Translation Lookaside Buffer

يعني إيه؟ Cache سريع بيحفظ آخر translations من page number → frame number
TLB Hit: لقى الـ page في الـ TLB → memory access واحدة بس
TLB Miss: مش موجود في TLB → يدور في الـ Page Table → memory accesses اتنين
Hit Ratio (α): نسبة الـ TLB Hits — أعلى = أسرع

EAT Calculator 🧮

EAT = (1+ε)α + (2+ε)(1-α) = 2 + ε - α حيث: ε = TLB lookup time α = Hit Ratio ma = Memory Access Time (1 unit)

Memory Protection في Paging

Valid-Invalid Bit: كل entry في الـ Page Table عنده bit:
Valid (v): الـ page موجودة في الـ logical address space للـ process وبشكل قانوني
Invalid (i): الـ page مش في الـ address space — لو process حاولت توصلها → Error
ليه مهمة؟

بتمنع process من الوصول لـ memory process تانية — الـ OS بيضمن Isolation بين الـ processes — ده أساس security في نظام الـ paging

ربط الدنيا ببعض

Process → Scheduling → Context Switch

الـ Process بيتكون ويدخل الـ Ready Queue

↓ الـ Short-term Scheduler بيختار الـ process وبعمل Context Switch

↓ الـ Process بتشتغل في الـ CPU

↓ لو عملت I/O بتروح Waiting Queue ← بيحصل Context Switch تاني

↓ لما I/O يخلص ترجع Ready Queue

System Call → Kernel Mode → Privilege

الـ User Program محتاج يعمل I/O → يعمل System Call

↓ الـ CPU بيتحول من User Mode → Kernel Mode

↓ الـ Kernel بيعمل العملية المطلوبة

↓ بعد الخلاص الـ CPU يرجع Kernel → User Mode

✅ ده بيحمي الـ system — الـ user مش بيقدر يعمل privileged instructions مباشرة

Paging → Fragmentation → Memory Protection

الـ Paging بيقضي على External Fragmentation

↓ بس ممكن يسبب Internal Fragmentation في آخر page

↓ كل entry في الـ Page Table عنده valid/invalid bit

↓ ده بيعمل Memory Protection — process مش تقدر توصل لـ memory process تانية

TLB → EAT → Performance

الـ Page Table في الـ memory → كل access محتاجة 2 memory accesses

↓ الـ TLB بيشتغل كـ cache → لو Hit → memory access واحدة بس

↓ الـ EAT بيحسب متوسط الوقت حسب Hit Ratio

↓ كل ما Hit Ratio ↑ → EAT قرب من memory access time واحد

⚡ Scheduling Summary

FCFS: أبسط، convoy effect، non-preemptive

SJF: أقل avg WT، starvation، non-preemptive

SRTF: SJF preemptive، أحسن avg WT حقيقياً

RR: fair، no starvation، preemptive، time quantum

Preemptive: ممكن تاخد CPU في النص

Non-preemptive: لازم تكمل أو تعمل I/O

Starvation: process مش بتاخد CPU ابداً

Convoy Effect: processes تطيلة بتحبس التقيلة

❌ Interrupt = Trap → غلط! Interrupt من HW، Trap من SW

❌ Ready = Waiting → غلط! Ready = في طابور CPU، Waiting = مستني I/O

❌ TLB Miss = Page Fault → غلط! TLB Miss = بدور في page table، Page Fault = مش في memory خالص

❌ Paging بتحل internal fragmentation → غلط! بتحل external

❌ SJF مش preemptive → صح لـ SJF، لكن SRTF هو الـ preemptive version
