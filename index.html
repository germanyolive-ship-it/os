📖OS Basics — أهم تعريفات
  • 🔸 OS = intermediary بين user وال hardware
  • 🔸 Kernel = القلب الحقيقي لل OS، اللي بيتحمل في memory طول الوقت
  • 🔸 OS ≠ Kernel — OS أكبر، Kernel جزء منه
  • 🔸 System Call = الطريقة الوحيدة إن الـ user program يطلب خدمة من الـ Kernel
  • 🔸 User Mode → Kernel Mode بيتعمل عن طريق System Call أو Interrupt
  • 🔸 Interrupt = من الـ hardware (خارجي) | Trap = من الـ software (داخلي/user)
  • 🔸 Bootstrap = أول برنامج بيشتغل لما الكمبيوتر يبدأ (في ROM)

⚙️Process — أهم نقاط
  • 🔸 Process = program قيد التنفيذ (active) | Program = ملف ساكن (passive)
  • 🔸 States: New → Ready → Running → Waiting → Terminated
  • 🔸 PCB = Process Control Block — فيه كل info عن الـ process
  • 🔸 Context Switch = لما الـ CPU يتنقل من process لـ process — بيحفظ/يرجع PCB
  • 🔸 Long-term scheduler: disk → memory | Short-term: memory → CPU
  • 🔸 Running → Waiting: بسبب I/O | Running → Ready: بسبب interrupt
📊Scheduling Formulas 🔢
Turnaround Time (TAT):
= Completion Time − Arrival Time
Waiting Time (WT):
= TAT − Burst Time
Response Time:
= First Response − Arrival Time
FCFS: أسرع في تطبيق | SJF: أقل avg WT | RR: أفضل response time

Overview — أساسيات OS

ما هو الـ Operating System؟

يعني إيه؟

الـ OS هو برنامج بيعمل كـ intermediary (وسيط) بين الـ user والـ hardware — بيسهل الاستخدام وبيدير الموارد

كـ Resource Allocator

بيوزع موارد الكمبيوتر (CPU، memory، I/O) بشكل عادل وكفء بين البرامج

كـ Control Program

بيتحكم في تنفيذ البرامج عشان يمنع الأخطاء وسوء الاستخدام

🎯 Goals of OS

  • ✅ تنفيذ برامج الـ user وحل مشاكله
  • ✅ الكمبيوتر يكون سهل الاستخدام
  • ✅ استخدام الـ hardware بكفاءة
  • ✅ حماية البيانات والموارد (Protection)
  • ✅ التواصل بين البرامج المختلفة

الـ Kernel 🔑

ما هو الـ Kernel؟

هو القلب الحقيقي لل OS — أول حاجة بتتحمل في memory وقت الـ Boot وبيفضل موجود طول الجلسة

وظائفه:
  • • Scheduler: بيوزع وقت CPU
  • • Memory Manager: بيدير الـ RAM
  • • Interrupt Handler: بيعالج الـ interrupts
  • • Supervisor: بيدير الصلاحيات
⚠️ مش نفس الـ OS!

الـ OS أكبر — بيشمل Kernel + Utilities + Drivers + User Interface

الـ Kernel بس هو اللي بيشتغل في Kernel Mode

System Calls 📞

يعني إيه؟

الطريقة الوحيدة اللي الـ user program بيستخدمها عشان يطلب خدمة من الـ Kernel (زي فتح ملف، عمل process جديد، إلخ)

🔸 بيتسمى كمان software interrupt

🔸 لما الـ system call بيتنادى → الـ mode بيتغير من User → Kernel

🔸 بعد ما الـ Kernel ينهي الخدمة → بيرجع Kernel → User Mode

⚡ Interrupt vs Trap — فرق مهم جداً

Interrupt (Hardware Interrupt)
  • ✅ مصدره: الـ Hardware (keyboard, disk, timer)
  • ✅ خارجي — من جهاز
  • ✅ بيحصل في أي وقت (asynchronous)
  • ✅ مثال: الـ disk انتهى من قراءة بيانات
Trap (Software Interrupt)
  • ✅ مصدره: الـ Software (برنامج المستخدم)
  • ✅ داخلي — من instruction
  • ✅ بيحصل بشكل متعمد أو بسبب خطأ
  • ✅ مثال: System Call أو Division by Zero

🎯 امتحانياً: كلاهما بيخلي الـ CPU يتحول لـ Kernel Mode ويشغل handler معين

OS Services

User Interface (UI): CLI أو GUI للتواصل مع المستخدم
Program Execution: تحميل وتشغيل البرامج
I/O Operations: التعامل مع أجهزة الإدخال/الإخراج
File System Manipulation: قراءة/كتابة/حذف ملفات
Communication: التواصل بين الـ processes
Error Detection: اكتشاف ومعالجة الأخطاء
Resource Allocation: توزيع CPU, Memory, I/O
Protection & Security: حماية الموارد والبيانات

Booting Process 🚀

1
Power ON → CPU ينشط الـ BIOS/UEFI
2
POST (Power-On Self-Test) → بيفحص الـ Hardware كله
3
بيقرأ الـ MBR (Master Boot Record) من الـ disk
4
Bootstrap Loader (في ROM) بيحمل الـ Kernel في الـ RAM
5
الـ Kernel بيشتغل ويتحكم في كل حاجة
⚠️ الـ Bootstrap program عادة بيتخزن في ROM أو EEPROM — مش في RAM

أنواع الـ Operating Systems

📦

Batch Systems

يعني إيه؟ Jobs بتتجمع وتشتغل واحدة ورا التانية بدون تدخل من المستخدم
العيب: لو job واحدة وقعت في I/O → الـ CPU بتقف (idle) — مش كفاءة!
الميزة: بسيطة جداً في التنفيذ
🔄

Multiprogramming

يعني إيه؟ أكتر من process في الـ memory في نفس الوقت — لما واحدة تعمل I/O، التانية تشتغل
الفايدة: CPU مش بتقف — استخدام أحسن
ملاحظة: مفيش حاجة اسمها "conversation" مع المستخدم
⏱️

Time-Sharing

يعني إيه؟ نوع من الـ Multiprogramming — بس الـ CPU بتتبدل بسرعة بين الـ users عن طريق time slices
الميزة: كل user حاسس إنه عنده الـ CPU لوحده (interactive)
امتحانياً: الفرق بين Multiprogramming وTime-Sharing هو وجود time quantum والـ interactivity
⚡

Real-Time Systems

يعني إيه؟ لازم الـ response يحصل في وقت محدد (deadline) — التأخير ممكن يكون كارثي
أمثلة: طائرات، أنظمة طبية، محطات نووية

نوعان:

• Hard Real-Time: فشل الـ deadline = كارثة

• Soft Real-Time: فشل الـ deadline مش نهاية الدنيا

🌐

Distributed Systems

يعني إيه؟ أكتر من كمبيوتر مترابطين عن طريق network ويشتغلوا كـ system واحد
الميزة: زيادة في الـ reliability والـ performance
📱

Embedded Systems

يعني إيه؟ OS مخصص لجهاز معين — بيعمل مهام محددة بس
خصائص: Kernel صغير + real-time + محدود الموارد
أمثلة: موبايل، سيارة، ميكروويف، راوتر
🖥️

Virtual Machines

يعني إيه؟ Software بيعمل emulation لـ machine كاملة — بتخلي OS يشتغل فوق OS تاني
Process VM: بيعمل environment لـ process واحدة (زي JVM)
System VM: بيعمل environment لـ OS كامل (زي VMware, VirtualBox)
⚠️ امتحانياً: الـ VM بيديك illusion إنك عندك machine منفصلة — في الحقيقة Hardware واحد مقسم
فايدته: تشغيل أكتر من OS على نفس الجهاز — isolation و testing و cloud computing
⚙️

Context Switch

يعني إيه؟ لما الـ CPU تتنقل من process لـ process تانية — بتحفظ PCB القديمة وتحمل PCB الجديدة
1 Save state of current process في الـ PCB بتاعته
2 Select next process من الـ ready queue
3 Load state of new process من الـ PCB بتاعته
4 Resume execution من النقطة اللي وقف عندها
⚠️ Cost: Context switch بياخد وقت (overhead) — خلال الـ switch، مفيش شغل حقيقي بيحصل

أنواع الـ Schedulers

Schedulerبيشتغل بينوظيفتهسرعة الاستدعاء
Long-term (Job Scheduler) Disk → Memory بيقرر أنهي processes تتحمل في الـ RAM بطيء (دقائق)
Short-term (CPU Scheduler) Memory → CPU بيقرر أنهي process تاخد الـ CPU دلوقتي سريع جداً (ms)
Medium-term Memory ⇄ Disk Swapping — بيشيل processes من memory متوسط
امتحانياً: اللي بيتكلم عنه أكتر في الامتحان هو الـ Short-term scheduler (CPU Scheduler) — ده اللي بيعمل الـ FCFS, SJF, RR

Scheduling Queues

Job Queue

كل الـ processes في النظام (disk + memory)

Ready Queue

الـ processes اللي في الـ memory وجاهزة تشتغل في الـ CPU

Device Queue

الـ processes اللي بتستنى جهاز I/O يخلص

📊

CPU Scheduling

Scheduling Criteria — بنحكم على الـ Algorithm بإيه؟

CPU Utilization ↑ نسبة استخدام CPU — أعلى أفضل
Throughput ↑ عدد الـ processes المنتهية في الوحدة الزمنية
Turnaround Time ↓ الوقت من بدء لإنهاء العملية
Waiting Time ↓ الوقت في الـ ready queue
Response Time ↓ وقت أول استجابة للـ request
Fairness ↑ كل process تاخد حظها

Preemptive vs Non-preemptive

Non-preemptive: لما process تاخد الـ CPU، مفيش حاجة تقدر تاخدها منها إلا لو هي خلصت أو طلبت I/O
Preemptive: ممكن تاخد الـ CPU من process وتديها لحد تاني في النص — أعدل وأحسن response time
FCFS

First-Come First-Served

الفكرة: الأول يجي الأول يتخدم — زي طابور عادي
العيب: Convoy Effect — process طويلة ممكن تخلي كل التانيين يستنوا
الميزة: بسيطة جداً في التطبيق
نوع: Non-preemptive فقط
SJF/SRTF

Shortest Job First / SRTF

الفكرة: الـ process اللي Burst Time أقل تشتغل الأول
الميزة: أقل average waiting time نظرياً (optimal)
العيب: Starvation — الـ long processes ممكن تستنى للأبد
SRTF = SJF Preemptive — لو جت process أقل burst time، بتاخد الـ CPU
RR

Round Robin

الفكرة: كل process بتاخد time quantum (q) — لو مخلصتش، ترجع للآخر وتستنى
الميزة: عادل جداً — لا Starvation — أحسن response time
التحدي: لو q صغير → context switches كتير. لو q كبير → بيبقى زي FCFS
نوع: Preemptive

Memory Management

(حتى Slide 40 بس)

Logical vs Physical Address

Logical Address (Virtual)

العنوان اللي الـ CPU بيولده — الـ program بيشوفه

مش العنوان الحقيقي في الـ RAM

MMU

Memory Management Unit — Hardware بيحول Logical → Physical

بيضيف Relocation Register على الـ Logical Address

Physical Address

العنوان الحقيقي في الـ RAM

الـ memory unit بتشوفه

Address Binding — متى بتتحدد العناوين؟

Compile Time

لو عارفين مكان الـ process في الـ memory من الأول — الـ compiler بيولد physical addresses مباشرة

Load Time

لما الـ OS بيحمل البرنامج في الـ memory — بيعمل relocatable code

Execution Time ✅ الأشيع

العنوان بيتحدد أثناء التشغيل — بيحتاج MMU hardware — الـ process ممكن تتحرك في الـ memory

Contiguous Allocation وطرق الـ Allocation

Contiguous: الـ process بتاخد block متواصل في الـ memory
First-Fit: خد أول Hole كبير كفاية — أسرع
Best-Fit: خد أصغر Hole كافي — بيقلل الهدر بس أبطأ
Worst-Fit: خد أكبر Hole — بيسيب أكبر Hole متبقي
امتحانياً: First-fit وBest-fit أحسن من Worst-fit في الغالب

Fragmentation — أهم فرق في الميموري

External Fragmentation

المساحة الكلية الفاضية كافية للـ process، بس مش متواصلة (متقطعة)

بيحصل في Contiguous Allocation

الحل: Compaction أو Paging

Internal Fragmentation

المساحة المخصصة للـ process أكبر من اللي هي محتاجاه — الفرق waste

بيحصل في Paging (آخر page مش كاملة)

Paging 📄

الفكرة: الـ Physical Memory بتتقسم لـ Frames (أحجام ثابتة) — الـ Logical Memory بتتقسم لـ Pages (نفس الحجم)
الميزة: بتحل External Fragmentation — الـ process مش لازم تكون متواصلة في الـ memory
العيب: Internal Fragmentation في آخر page
Page Table: جدول بيربط كل page رقم كام بالـ frame اللي فيها
Logical Address = Page Number | Offset Page Number (p) → index في الـ Page Table Offset (d) → مكانها داخل الـ Frame Physical Address = Frame Base + Offset لو Logical Address Space = 2^m و Page Size = 2^n → Page Number = m-n bits → Offset = n bits

🔢 Paging Address Translator

أدخل البيانات وكبس Translate

TLB — Translation Lookaside Buffer

يعني إيه؟ Cache سريع بيحفظ آخر translations من page number → frame number
TLB Hit: لقى الـ page في الـ TLB → memory access واحدة بس
TLB Miss: مش موجود في TLB → يدور في الـ Page Table → memory accesses اتنين
Hit Ratio (α): نسبة الـ TLB Hits — أعلى = أسرع

EAT Calculator 🧮

EAT = (1+ε)α + (2+ε)(1-α) = 2 + ε - α حيث: ε = TLB lookup time α = Hit Ratio ma = Memory Access Time (1 unit)

Memory Protection في Paging

Valid-Invalid Bit: كل entry في الـ Page Table عنده bit:
Valid (v): الـ page موجودة في الـ logical address space للـ process وبشكل قانوني
Invalid (i): الـ page مش في الـ address space — لو process حاولت توصلها → Error
ليه مهمة؟

بتمنع process من الوصول لـ memory process تانية — الـ OS بيضمن Isolation بين الـ processes — ده أساس security في نظام الـ paging

Process Synchronization

Race Condition

التعريف: لما أكتر من process/thread بيوصلوا وبيعدّلوا على Shared Data في نفس الوقت، والنتيجة النهائية بتعتمد على ترتيب التنفيذ (timing)
ليه بتحصل؟ لأن العمليات على المتغير المشترك (زي count++) مش Atomic — بتتحول لعدة تعليمات (load, add, store)
النتيجة: قيمة غلط أو غير متوقعة للمتغير المشترك — bug صعب اكتشافه لأنه بيظهر بالصدفة
// Shared variable: count = 5 // Thread A Thread B register1 = count register2 = count register1++ register2++ count = register1 count = register2 // لو الترتيب: A load → B load → A store → B store // النتيجة count = 6 (المفروض تكون 7!)

Critical Section

التعريف: جزء من الكود فيه access لـ shared resource (variable, file, data structure) ولازم ما يتنفذش من أكتر من process في نفس الوقت
General Structure: Entry Section → Critical Section → Exit Section → Remainder Section

Critical Section Problem — 3 شروط

1. Mutual Exclusion: process واحدة بس تقدر تكون داخل الـ Critical Section في نفس اللحظة
2. Progress: لو مفيش process داخل الـ CS وفيه processes عايزة تدخل، الاختيار بينهم لازم يحصل (مش انتظار للأبد)
3. Bounded Waiting: فيه حد أقصى لعدد المرات اللي processes تانية تدخل الـ CS قبل ما طلب process معينة يتقبل (منع starvation)

Locks (Mutex Locks)

الفكرة: الـ process لازم تـ acquire() الـ lock قبل ما تدخل الـ Critical Section وتعمل release() لما تخلص
do { acquire lock critical section release lock remainder section } while (true);
مهم: لو الـ acquire/release نفسهم مش atomic، الـ Mutual Exclusion مش هتكون مضمونة!

Spinlock

التعريف: نوع من الـ lock فيه الـ process بتفضل في loop (busy-wait) تتأكد من الـ lock لحد ما تبقى متاحة
الميزة: مفيش context switch overhead — مفيد لو الانتظار قصير ومفيش CPU كتير
العيب: بيستهلك CPU cycles وهو مستني (busy waiting) — سيئ لو الانتظار طويل أو على Single CPU

Test-and-Set Instruction

التعريف: Hardware instruction atomic — بترجع القيمة القديمة للمتغير وفي نفس الوقت بتخليه true
ليه atomic؟ الـ CPU بيضمن إن مفيش process تانية تقدر تتدخل وسط التنفيذ — العملية كلها instruction واحدة hardware-level
boolean test_and_set(boolean *target) { boolean rv = *target; *target = true; return rv; } // استخدامه كـ lock: do { while (test_and_set(&lock)) ; // busy wait // critical section lock = false; // remainder section } while (true);

⚖️ Peterson's Algorithm

الفكرة: حل software بيحقق الـ 3 شروط (Mutual Exclusion, Progress, Bounded Waiting) لـ 2 processes فقط (P0, P1)
flag[i]: بتقول إن process i عايزة تدخل الـ Critical Section
turn: رقم الـ process اللي دورها تدخل (لو الاتنين عايزين يدخلوا)
الشرط للدخول: process Pi بتدخل لو flag[j] == false أو turn == i
// Process Pi (j = الـ process التانية) do { flag[i] = true; turn = j; while (flag[j] && turn == j) ; // critical section flag[i] = false; // remainder section } while (true);
امتحانياً: Peterson's Algorithm شغال نظرياً لـ 2 processes فقط، وعلى المعالجات الحديثة مش مضمون 100% بسبب reordering في الـ memory (compiler/CPU optimizations)

Readers-Writers Problem

الفكرة: Shared data (زي database) عليها processes نوعين: Readers (بيقروا بس) و Writers (بيقروا ويكتبوا)
القاعدة 1: أكتر من Reader يقدروا يقروا في نفس الوقت (Read مش بيغير الـ data)
القاعدة 2: الـ Writer لازم يكون عنده exclusive access — مفيش Reader أو Writer تاني وقتها
الحلول الشائعة:

First Readers-Writers: الأولوية للـ Readers — ممكن يحصل Writer Starvation

Second Readers-Writers: الأولوية للـ Writers — ممكن يحصل Reader Starvation

الحل عادة بيستخدم Semaphores: mutex و wrt وعداد readcount

Deadlocks

تعريف Deadlock

مجموعة من الـ processes كل واحدة فيها مستنية event (عادة الإفراج عن resource) ممسوك من process تانية في نفس المجموعة — النتيجة: كل العمليات stuck للأبد ومش هتقدر تتقدم

4 شروط ضرورية للـ Deadlock (لازم الأربعة معًا)

1. Mutual Exclusion: الـ resource مش ممكن تتشارك — process واحدة بس تستخدمها في كل لحظة
2. Hold and Wait: الـ process بتمسك resource واحدة على الأقل وفي نفس الوقت بتطلب resources تانية ممسوكة من processes تانية
3. No Preemption: الـ resource مش ممكن تتاخد بالقوة من الـ process — لازم تسيبها بنفسها بعد ما تخلص
4. Circular Wait: فيه مجموعة دائرية من processes، كل واحدة مستنية resource ممسوكة من اللي بعدها في الدائرة
امتحانياً: لو شرط واحد بس مش متحقق → مفيش Deadlock. كل أساليب الـ Prevention بتكسر شرط واحد من الأربعة

Resource Allocation Graph (RAG)

العناصر: Vertices اتنين: Processes (دوائر) و Resources (مستطيلات/نقط)
Request Edge (Pi → Rj): الـ process Pi طالبة resource Rj
Assignment Edge (Rj → Pi): resource Rj متاحة/ممسوكة من process Pi
القاعدة الأساسية:
• لو الـ graph ماعندوش Cycle → مفيش Deadlock
• لو فيه Cycle ومافيش instances متعددة للموارد → Deadlock مضمون
• لو فيه Cycle وفيه multiple instances للموارد → ممكن يكون Deadlock وممكن لا (Cycle شرط ضروري بس مش كافي)
مثال Cycle بسيط: P1 → R1 (P1 طالبة R1) R1 → P2 (R1 ممسوكة من P2) P2 → R2 (P2 طالبة R2) R2 → P1 (R2 ممسوكة من P1) Cycle: P1 → R1 → P2 → R2 → P1 لو كل resource عندها instance واحدة فقط → Deadlock مؤكد!

Prevention — كسر الشروط

كسر Mutual Exclusion: صعب — بعض الموارد (printer) لازم تكون exclusive، لكن ممكن نستخدم spooling لموارد قابلة للمشاركة
كسر Hold and Wait: الـ process تطلب كل الموارد اللي محتاجاها مرة واحدة في البداية، أو تسيب كل الموارد قبل ما تطلب جديدة
كسر No Preemption: لو process طالبة resource مش متاحة، الموارد اللي معاها تتاخد منها (preempted) وترجع لها بعدين
كسر Circular Wait: ترتيب كل الموارد بـ numbering، وكل process لازم تطلب الموارد بترتيب تصاعدي بس

Avoidance & Banker's Algorithm

الفكرة: الـ OS عنده معلومة مسبقة عن أقصى احتياجات كل process من كل resource — وبيقرر يوافق على الطلب بس لو النظام يفضل في Safe State
Safe State: فيه ترتيب (sequence) للـ processes كلها بحيث كل process تقدر تكمل لو نفذت بهذا الترتيب
Unsafe State: مش بالضرورة Deadlock، لكن ممكن يؤدي له
Data Structures:

Available: عدد الموارد المتاحة من كل نوع

Max: أقصى طلب لكل process من كل resource

Allocation: الموارد المخصصة حاليًا لكل process

Need = Max - Allocation: الباقي اللي ممكن تطلبه

Safety Algorithm: دور على process بحيث Need[i] ≤ Available — لو لقيتها افرض إنها خلصت وأضف Allocation[i] لـ Available — كرر لحد ما كل الـ processes تخلص

🏦 Banker's Algorithm — Safety Checker

مثال جاهز: 5 processes، 3 resource types (A,B,C). كبس Run للتحقق من Safe State وإيجاد Safe Sequence

Detection

لو الموارد single instance: استخدم Wait-For Graph (تبسيط للـ RAG بدون resource nodes) — Cycle = Deadlock
لو الموارد multiple instances: استخدم algorithm شبيه بـ Banker's بس بدل Max نستخدم Request matrix — لو فيه processes متبقية ما تقدرش تكمل → Deadlocked
دورية الفحص: الـ OS يقرر امتى يشغل detection algorithm — كل ما زاد عدد الـ deadlocks المتوقعة، كل ما زادت دورية الفحص

Recovery

1. Process Termination:

• Abort كل الـ processes المتورطة في الـ deadlock (مكلف)

• Abort process واحدة في كل مرة لحد ما يختفي الـ deadlock (أرخص، بس overhead إعادة فحص)

2. Resource Preemption:

• اختيار victim (أقل تكلفة)

• Rollback للـ process لحالة آمنة سابقة

• منع Starvation — process واحدة متختارش كـ victim كل مرة

⚠️ أفخاخ شائعة في Deadlock

❌ Cycle = Deadlock دايماً: غلط لو فيه multiple instances من resource — الـ cycle شرط ضروري بس مش كافي
❌ Banker's بيحل الـ Deadlock: غلط — Banker's بيـ تجنب الدخول في deadlock من الأساس، مش بيعالجه بعد حدوثه
❌ Safe State = مفيش مشاكل أبدًا: Safe State يعني فيه ترتيب آمن متاح، لكن مش بالضرورة كل ترتيب آمن
❌ Unsafe State = Deadlock فورًا: غلط — Unsafe State معناها ممكن يؤدي لـ Deadlock، لكن مش حتمي

Virtual Memory

الفكرة الأساسية

Virtual Memory: تقنية بتسمح بتنفيذ process حتى لو مش كل أجزائها موجودة في الـ Physical Memory في نفس الوقت — بتفصل بين الـ Logical Memory (اللي يشوفها الـ process) والـ Physical Memory الحقيقية
الفايدة:

• ممكن تشغل process أكبر من حجم الـ RAM الفعلي

• زيادة درجة الـ Multiprogramming (أكتر processes في الذاكرة)

• كل process بتحمل بس الأجزاء اللي محتاجاها فعلاً

Demand Paging

التعريف: الـ pages بتتحمل في الـ Memory فقط لما تحتاجها الـ process (Lazy Swapper) — مش كل الـ process بتتحمل مرة واحدة
الفايدة: أقل I/O، أقل memory مستخدمة، استجابة أسرع، أكتر users يشاركوا
Pure Demand Paging: لا يتم تحميل أي page إلا عند الحاجة الفعلية (حتى الأول page بتتحمل بـ Page Fault)
Demand Paging Steps: 1. Process بتحاول توصل لـ page 2. CPU يشوف الـ valid/invalid bit 3. لو invalid → Page Fault Trap 4. OS يشوف لو reference صحيح - لو غير صحيح → terminate process - لو صحيح → يكمل (الخطوات التالية)

Page Fault وخطوات معالجته

التعريف: Trap بيحصل لما process تحاول توصل لـ page موجودة في الـ Logical Address Space بس مش موجودة في الـ Physical Memory دلوقتي (valid bit = invalid)
Page Fault Handling — الخطوات: 1. تشييك الـ internal table (في PCB) عشان نعرف لو الـ reference صحيح أو غلط 2. لو غلط → terminate process 3. لو صحيح بس الـ page مش محمّلة: a. دور على free frame (من free-frame list) b. اطلب الـ page من الـ disk c. عند الانتهاء، حدّث الـ Page Table (set valid bit = v) 4. أعد تنفيذ الـ instruction اللي عملت الـ fault

Valid / Invalid Bit

Valid (v): الـ page موجودة في الـ Physical Memory دلوقتي — الوصول ليها مباشر
Invalid (i): أحد احتمالين: (1) الـ page مش جايه على الـ Memory أصلاً (Page Fault) أو (2) reference غير شرعي (الـ process مالهاش حق توصل للـ page دي)
الـ Page Table بيستخدم نفس الـ bit ده للأمان (Memory Protection) ولـ Demand Paging مع بعض

Address Translation في VM

نفس آلية الـ Paging: Logical Address = Page Number + Offset
إضافة: Page Table Entry فيه bit إضافي اسمه Valid/Invalid — لو valid، الـ frame number صحيح ونحول مباشرة، لو invalid، يحصل Page Fault Trap
Backing Store: منطقة على الـ Disk (Swap Space) بتحتفظ بالـ pages اللي مش موجودة في الـ Memory دلوقتي

🧮 Effective Access Time مع Page Faults

Backing Store: الـ Disk هو المكان اللي تتخزن فيه الـ pages اللي اتشالت من الـ memory — الوصول له بطيء جدًا (milliseconds) مقارنة بالـ memory (nanoseconds)
EAT = (1 - p) × ma + p × page_fault_time حيث: p = Page Fault Rate (0 ≤ p ≤ 1) ma = Memory Access Time page_fault_time = وقت معالجة الـ Page Fault (بيشمل disk I/O — بطيء جدًا)
مثال: ma = 200ns، page fault time = 8ms = 8,000,000ns، p = 0.001

EAT = (1-0.001)×200 + 0.001×8,000,000
= 199.8 + 8000 = 8199.8 ns

الخلاصة: حتى نسبة Page Fault صغيرة جدًا (0.1%) بتأثر بشكل كبير على EAT — لازم نقلل p لأقصى حد

Programmed I/O

الفكرة: الـ CPU بيـ poll busy-wait status register للـ device لحد ما يكون جاهز، وبعد كل byte/word بيتأكد تاني
العيب: الـ CPU مش بيعمل أي شيء مفيد وقت الانتظار — busy waiting يستهلك CPU cycles بالكامل

Interrupt-Driven I/O

الفكرة: الـ CPU بيـ initiate الـ I/O ويكمل شغل process تانية، لما الـ device يخلص بيبعت Interrupt للـ CPU عشان يتعامل مع البيانات
الميزة: الـ CPU مش busy-waiting — بيشتغل على processes تانية وقت ما الـ I/O جاري
العيب: لسه فيه context switch overhead لكل byte/word في data transfer كبير

DMA — Direct Memory Access

الفكرة: الـ CPU بيعطي الـ DMA Controller معلومات (source, destination, عدد البايتات) ويبدأ العملية، والـ DMA Controller بينقل البيانات مباشرة بين الـ Device والـ Memory بدون تدخل الـ CPU
الـ Interrupt الوحيد: الـ DMA Controller بيـ interrupt الـ CPU مرة واحدة فقط لما تنتهي العملية كلها (block كامل، مش byte by byte)
مستخدم في:

• High-speed I/O devices (Disk, Network Card)

• نقل كميات كبيرة من البيانات

الفايدة الكبرى: الـ CPU بيبقى متفرغ تمامًا للمعالجة طول فترة نقل البيانات

Single Buffer vs Double Buffer

Single Buffering:

يوجد buffer واحد فقط — الـ Device بيكتب فيه والـ CPU بيقرا منه

المشكلة: الـ Device لازم يستنى لحد ما الـ CPU يفضي الـ buffer قبل ما يكتب فيه تاني — لا يوجد تداخل (overlap) بين عمليتي القراءة والمعالجة

Double Buffering:

يوجد بافرين — الـ Device بيكتب في buffer واحد بينما الـ CPU بيعالج محتوى الـ buffer التاني، وبعدين يتبادلوا الأدوار

الفايدة: overlap بين I/O والمعالجة → throughput أعلى

امتحانياً: Double Buffering ≠ ينهي مشكلة الانتظار بالكامل لو سرعة الإنتاج والاستهلاك مختلفة جدًا — بس بيقللها بشكل كبير

Disk Structure

Platter: القرص الدائري المغناطيسي اللي عليه البيانات — الـ disk فيه أكتر من platter
Track: دائرة على الـ platter بيتم تخزين البيانات عليها
Sector: أصغر وحدة قابلة للقراءة/الكتابة على الـ track — حجمها ثابت (عادة 512 bytes)
Cylinder: مجموعة الـ tracks اللي في نفس المسافة من مركز كل الـ platters (نفس الموضع على كل الأقراص)
Logical Block Addressing (LBA): الـ sectors الفعلية بتتحول لـ sequence واحدة من logical blocks مرقمة من 0 لـ N-1 — ده اللي يشوفه الـ OS بدل التعامل المباشر مع cylinder/track/sector

⏱️ Disk Access Time = Seek + Rotational Latency + Transfer

Seek Time

الوقت اللازم لتحريك الـ Read/Write Head ليوصل للـ Cylinder/Track المطلوب — أكبر عنصر في وقت الوصول عادة

Rotational Latency

الوقت اللازم للقرص (Platter) عشان يدور لحد ما الـ Sector المطلوب يوصل تحت الـ Head

Transfer Time

الوقت الفعلي لنقل البيانات بعد ما الـ Head بقى فوق الـ Sector الصحيح

Access Time = Seek Time + Rotational Latency + Transfer Time ملحوظة: Seek Time غالبًا أكبر العناصر — كل خوارزميات Disk Scheduling هدفها الأساسي تقليل Seek Time (عبر تقليل إجمالي حركة الـ Head)

ربط الدنيا ببعض

Process → Scheduling → Context Switch

الـ Process بيتكون ويدخل الـ Ready Queue

↓ الـ Short-term Scheduler بيختار الـ process وبعمل Context Switch

↓ الـ Process بتشتغل في الـ CPU

↓ لو عملت I/O بتروح Waiting Queue ← بيحصل Context Switch تاني

↓ لما I/O يخلص ترجع Ready Queue

System Call → Kernel Mode → Privilege

الـ User Program محتاج يعمل I/O → يعمل System Call

↓ الـ CPU بيتحول من User Mode → Kernel Mode

↓ الـ Kernel بيعمل العملية المطلوبة

↓ بعد الخلاص الـ CPU يرجع Kernel → User Mode

✅ ده بيحمي الـ system — الـ user مش بيقدر يعمل privileged instructions مباشرة

Paging → Fragmentation → Memory Protection

الـ Paging بيقضي على External Fragmentation

↓ بس ممكن يسبب Internal Fragmentation في آخر page

↓ كل entry في الـ Page Table عنده valid/invalid bit

↓ ده بيعمل Memory Protection — process مش تقدر توصل لـ memory process تانية

TLB → EAT → Performance

الـ Page Table في الـ memory → كل access محتاجة 2 memory accesses

↓ الـ TLB بيشتغل كـ cache → لو Hit → memory access واحدة بس

↓ الـ EAT بيحسب متوسط الوقت حسب Hit Ratio

↓ كل ما Hit Ratio ↑ → EAT قرب من memory access time واحد

🔑 تعريفات أساسية

OS: intermediary بين user وhardware

Kernel: أول ما بيتحمل عند Boot، بيفضل في memory

System Call: الطريق الوحيد من User → Kernel Mode

Process: program قيد التنفيذ (active)

PCB: data structure فيها كل info عن process

Context Switch: حفظ state + تحميل state تانية

MMU: Hardware بيحول logical → physical

TLB: cache للـ page table entries

Frame: وحدة في Physical Memory

Page: وحدة في Logical Memory

⚡ Scheduling Summary

FCFS: أبسط، convoy effect، non-preemptive

SJF: أقل avg WT، starvation، non-preemptive

SRTF: SJF preemptive، أحسن avg WT حقيقياً

RR: fair، no starvation، preemptive، time quantum

Preemptive: ممكن تاخد CPU في النص

Non-preemptive: لازم تكمل أو تعمل I/O

Starvation: process مش بتاخد CPU ابداً

Convoy Effect: processes تطيلة بتحبس التقيلة

⚠️ أكثر الأخطاء الشائعة

❌ OS = Kernel → غلط! OS أكبر

❌ Interrupt = Trap → غلط! Interrupt من HW، Trap من SW

❌ Ready = Waiting → غلط! Ready = في طابور CPU، Waiting = مستني I/O

❌ TLB Miss = Page Fault → غلط! TLB Miss = بدور في page table، Page Fault = مش في memory خالص

❌ Paging بتحل internal fragmentation → غلط! بتحل external

❌ SJF مش preemptive → صح لـ SJF، لكن SRTF هو الـ preemptive version
